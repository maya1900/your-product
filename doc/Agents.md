# Agents.md

## 1. 目的

本文档用于指导在本个人作品集网站中的 AI Agent。Agent 在实现、设计、内容整理和审查时，应以本文档为准，确保所有工作与项目目标一致。

本网站是个人作品集与个人品牌站，核心目标是：展示专业身份、呈现精选项目、说明服务能力、建立信任，并引导访客联系。

## 2. 产品理解

在修改任何内容前，Agent 需要先理解产品定位。

网站应呈现以下气质：

- 简洁
- 专业
- 有编辑感
- 现代
- 略带科技感
- 注重转化

参考方向包含强势的首屏、较大的标题、个人形象图、服务卡片、项目卡片、客户反馈以及结尾联系区。

不要直接照搬参考图，只借鉴其布局层级、留白和区块节奏。

## 3. 核心规则

### 3.1 必须遵守

- 保持页面简洁、清晰、精致。
- 优先保证内容表达，而不是花哨装饰。
- 业务内容要放在 data 或 content 文件中，不要散落在组件里。
- 保持良好的移动端适配。
- 使用语义化 HTML 和清晰的对比度。
- 依赖保持最少且有理由。
- 尽量优先静态渲染。
- 保持 SEO 质量。

### 3.2 禁止事项

- 未经明确允许，不要引入大型 UI 框架。
- 不要把作品集数据硬编码到多个 JSX 文件中。
- 不要添加超出 PRD 范围的功能。
- 不要为小型作品集站点过度设计抽象。
- 不要为了视觉效果牺牲性能。
- 不要破坏移动端布局。

## 4. 推荐技术栈

除非用户另有要求，否则优先使用：

- React
- TypeScript
- Vite
- Tailwind CSS
- Framer Motion（仅用于少量动效）
- Lucide React（图标）
- React Hook Form + Zod（如实现完整联系表单）
- Vercel（部署）

## 5. 项目结构建议

推荐结构：

```txt
src/
  components/
    layout/
    ui/
  sections/
  data/
  lib/
  types/
public/
  images/
```

Agent 应将内容放在合适的位置：

- 页面级内容放在 `src/sections/` 或页面入口。
- 可复用 UI 放在 `src/components/ui/`。
- 布局组件放在 `src/components/layout/`。
- 作品集内容放在 `src/data/`。
- 公共类型放在 `src/types/`。
- 工具函数放在 `src/lib/`。

## 6. 内容规则

### 6.1 内容来源

作品集内容必须数据化管理，可参考如下文件：

- `src/data/profile.ts`
- `src/data/projects.ts`
- `src/data/contact.ts`

### 6.2 个人内容

个人信息至少应包含：

- 姓名
- 身份定位
- 一句话简介
- 个人简介
- 头像或形象图路径
- 联系方式

### 6.3 项目内容

每个项目至少应包含：

- 标题
- 简介
- 分类
- 封面图
- 技术栈
- 结果或价值
- Demo、GitHub 或案例链接（如有）

### 6.4 联系内容

联系区域至少应包含：

- 联系引导语
- 邮箱
- 社交链接
- 表单或外链入口

## 7. 组件规则

### 7.1 页面区块

页面只保留以下核心区块：

- `Home` / 首页首屏
- `About` / 关于我
- `Projects` / 项目展示
- `Contact` / 联系方式

不要把不相关的内容合并进一个大组件。

### 7.2 UI 组件

复用组件应保持简单、可组合：

- `Button`
- `Badge`
- `Card`
- `Container`
- `SectionHeading`

除非确实能减少重复，否则不要加深层抽象。

### 7.3 客户端组件

仅在必要时使用客户端组件，例如：

- 动效包装
- 移动端导航状态
- 联系表单状态
- 简单计数动画

纯展示内容尽量保持为静态渲染。

## 8. 设计规则

### 8.1 视觉方向

视觉风格应使用：

- 大而有力量的标题
- 充足留白
- 清晰的层级关系
- 少量配色
- 轻微渐变或纹理
- 柔和边框
- 克制阴影

### 8.2 颜色

优先使用浅色背景、深色文字和单一强调色系。

建议色彩方向：

- 白色或接近白色的背景
- 深灰蓝文字
- 浅灰边框
- 蓝色或蓝紫色强调

### 8.3 字体

- 首页标题要足够突出。
- 各区块标题保持统一。
- 正文必须清晰可读，不要过小。
- 不要使用过多字体。

### 8.4 动效

动效应服务内容，而不是干扰内容。

可接受的动效：

- 区块渐显
- 卡片悬浮
- 按钮过渡
- Hero 图片轻微浮动
- 数字计数

避免重粒子、强视差和影响阅读的动效。

## 9. 响应式规则

### 9.1 桌面端

- 允许多栏布局。
- Hero 和项目区可以有更丰富的排版。
- 控制最大内容宽度。

### 9.2 平板端

- 减少网格列数。
- 保持卡片可读。
- 保留清晰的间距层级。

### 9.3 移动端

- 使用单列布局。
- CTA 按钮要易点击。
- 禁止横向滚动。
- 文本行长要舒适。
- 图片在小屏下仍要有意义。

## 10. SEO 规则

Agent 应维护或提升以下内容：

- 页面标题
- meta description
- Open Graph 信息
- Twitter 信息
- 标题层级语义
- 图片 alt 文本
- sitemap 和 robots 支持（如有）

如果新增项目详情页，每个页面都应有独立 metadata。

## 11. 可访问性规则

- 使用语义化 HTML。
- 确保键盘可操作。
- 表单字段必须有 label。
- 图片要有有意义的 alt。
- 保持足够的颜色对比度。
- 尊重减少动态效果的系统设置。

## 12. 性能规则

- 优化图片。
- 非首屏内容懒加载。
- 避免大型依赖。
- 动效保持轻量。
- 不要过早加载统计或第三方脚本。
- 优先静态渲染作品集内容。

## 13. 验证要求

完成有意义的改动后，Agent 需要检查：

- 项目能成功构建。
- TypeScript 没有新增错误。
- 没有明显的 lint 问题。
- 桌面端和移动端布局正常。
- 导航锚点可用。
- 如有联系表单，交互正常。
- SEO 配置有效。

只有在确实能降低回归风险时，才添加自动化测试。

## 14. 实施流程

推荐流程：

1. 先阅读 PRD 和 TECH_DESIGN。
2. 定义或更新数据模型。
3. 搭建页面结构和区块。
4. 补充样式和响应式布局。
5. 加入轻量交互。
6. 检查可访问性和 SEO。
7. 运行可用检查。
8. 清楚总结修改内容。

## 15. 输出要求

完成任务时，Agent 应总结：

- 改了什么
- 修改了哪些文件
- 为什么这样改
- 用户如何验证结果
- 是否有后续建议

## 16. 范围控制

MVP 只做一个精致的单页作品集站点。除非用户明确要求，不要加入博客系统、CMS、登录、支付、仪表盘或复杂后端。

未来可扩展为：

- 项目详情页
- 博客或笔记
- 简历页
- CMS 内容管理
- 多语言
- 服务报价页
